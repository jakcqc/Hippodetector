# App Design & Architecture

## Overview

The Hippodetector frontend is a multi-page **Streamlit** application (`server/`) that provides interactive views into congressional voting data, member stances, and (forthcoming) contradiction reports. It is read-only — all data is pre-generated by the offline pipeline and loaded at runtime from JSON files and Qdrant.

---

## Directory Structure

```
server/
├── app.py                    # Landing page + shared utilities + background preloader
└── pages/
    ├── 0_Press_Releases.py   # Browse scraped press releases
    ├── 1_Voted_Bills.py      # Explore bills with recorded votes
    ├── 2_Stances.py          # View per-member issue profiles + LLM analysis
    ├── 3_Embeddings_Map.py   # UMAP visualization of member embeddings
    └── 4_Backend.py          # System architecture diagram
```

Streamlit's file-based routing maps each `pages/N_Name.py` file to a sidebar navigation entry, in numerical order.

---

## Data Dependencies

The app reads from files generated by the offline pipeline. No live API calls are made during normal browsing.

| Data File | Used By | Description |
|---|---|---|
| `data/congress_members.json` | All pages | Member metadata (name, party, state, bioguideId) |
| `data/press_releases_by_bioguide.json` | Press Releases, Stances | Scraped press releases keyed by bioguideId |
| `data/congress_bills_voted_compact_last_10_years.json` | Voted Bills | Compact bill index with embedded vote records |
| `data/stances/{id}_issue_profile.json` | Stances | LLM-extracted stances from press releases |
| `data/stances/{id}_voting_issue_profile.json` | Stances | LLM-extracted stances from voting records |
| `data/press_release_embeddings_member_index.json` | Embeddings Map | Index of precomputed PR embeddings per member |
| `data/issue_profile_topic_embeddings/` | Embeddings Map | Topic-level embedding vectors |

Qdrant (vector DB) is used only by the contradiction detection pipeline, not directly by the UI yet.

## Shared Utilities (`app.py`)

`app.py` serves as the landing page and also exports helper functions imported by all page modules.

### Background Preloader

Large JSON files are loaded into a shared in-memory cache using a `ThreadPoolExecutor` at app startup:

```python
_PRELOAD_CACHE: dict[str, Any] = {}
_PRELOAD_LOCK = threading.Lock()
_PRELOAD_EXECUTOR = ThreadPoolExecutor(max_workers=2)
```

- Files are submitted for background loading when `app.py` first loads
- Pages call `get_preloaded_json(path)` which either returns the cached result or waits for the future to complete
- This hides I/O latency: by the time a user navigates to a page, the data is usually already in memory

### Key Shared Functions

| Function | Purpose |
|---|---|
| `load_json(path)` | Load JSON with preload cache fallback |
| `get_preloaded_json(path)` | Thread-safe access to preloaded data |
| `inject_css()` | Inject global CSS into every page |
| `normalize_text(text)` | Lowercase + strip for search normalization |
| `strip_html(html)` | Remove HTML tags for plain text display |

**UI layout:**

2. Result count
3. Paginated dataframe of matching press releases
4. Expandable detail view per release

### Page 1: Voted Bills

**File:** [server/pages/1_Voted_Bills.py](../server/pages/1_Voted_Bills.py)

The most data-heavy page. Allows exploration of all bills that received recorded votes over the past 10 years.

**Data source:** `data/congress_bills_voted_compact_last_10_years.json`

**Bill data shape:**
```
{
  billId, congress, billType, billNumber,
  title, summary (HTML), subjects,
  latestUpdateDate, versionCount,
  votes: [
    {
      chamber, question, result, voteDate, rollNumber,
      voteCounts: {yea, nay, present, notVoting},
      members: [{name, party, state, voteCast, bioguideId}]
    }
  ]
}
```

**Filtering pipeline** (applied in order):
1. Congress number
2. Chamber (House / Senate)
3. Bill result (Passed / Failed / Unknown / Mixed)
4. Proposer party
5. Proposer state
6. Fuzzy text search on title + summary + bill ID (threshold 0.42, `token_set_ratio`)

**Sort options:** Latest update, Vote count, Version count, Bill ID, Title

**Search implementation:**
- `bill_search_text()` builds a concatenated search string per bill
- `fuzzy_filter_indices()` uses `rapidfuzz.process.extract` for fast fuzzy matching
- `build_sorted_indices()` computes sort order separately from filtering so both can be combined

**UI layout:**
1. Sidebar with all filter controls + pagination config (1–250 per page)
2. Result count + page controls
3. Expandable bill cards showing:
   - Title, summary, subjects
   - Vote event table (date, chamber, result, vote counts)
   - Member vote list per event

### Page 2: Stances

**File:** [server/pages/2_Stances.py](../server/pages/2_Stances.py)

Displays per-member issue profiles derived from two sources: press releases and voting records. Also includes an LLM-powered analysis sidebar for on-demand comparison.

**Data sources:**
- `data/stances/{bioguideId}_issue_profile.json` — press release stances
- `data/stances/{bioguideId}_voting_issue_profile.json` — voting record stances

**Issue categories (24 total):**
```
abortion, budget_economy, civil_rights, corporations, crime, drugs,
education, energy_oil, environment, families_children, foreign_policy,
free_trade, government_reform, gun_control, health_care, homeland_security,
immigration, jobs, principles_values, social_security, tax_reform,
technology, war_peace, welfare_poverty
```

**Stance profile shape** (per member):
```
{
  bioguideId,
  stances: {
    public_release: {
      issues: { health_care: {summary, evidence}, ... }
    },
    voting_record: {
      issues: { health_care: {summary, evidence}, ... }
    }
  }
}
```
`evidence` is a count of supporting press releases or bills.

**Filtering:**
- Party, state
- Fuzzy search across member name and issue summaries

**UI layout:**
1. Sidebar: filters + member selector
2. Main area: side-by-side comparison of public release stances vs voting record stances for selected member
3. Issue grid with evidence counts

**LLM Analysis Sidebar (`render_llm_sidebar()`):**
- Model selector (GPT 5.2, Claude Sonnet 4.5)
- Custom prompt input
- On submit: sends member's two profiles to Archia API
- Prompt template:
  ```
  Compare public_issue_profile vs voting_issue_profile for this member.
  Respond in Markdown with sections:
    - Overall Assessment
    - Potential Contradictions
    - Alignments
    - Data Gaps
  ```
- Response streamed back and rendered as Markdown

This sidebar is the current closest approximation to contradiction detection while the full RAG pipeline is still being implemented.

### Page 3: Embeddings Map

**File:** [server/pages/3_Embeddings_Map.py](../server/pages/3_Embeddings_Map.py)

Visualizes member embeddings reduced to 2D/3D using UMAP. Allows exploration of which members cluster together semantically based on their press releases or issue topic profiles.

**Data sources:**
- `data/press_release_embeddings_member_index.json`
- `data/issue_profile_topic_embeddings/`

**Processing (on demand):**
1. Load embedding vectors for selected dataset
2. Run UMAP dimensionality reduction (2D or 3D)
3. Render interactive Plotly scatter plot
4. Color by party, hover tooltip shows member name + metadata

**Dataset options:** Press releases | Issue topic profiles

Heavy dependencies (Plotly, UMAP, pandas, numpy) are imported lazily on this page only to avoid slowing down other pages.

## UI Patterns

### Caching Strategy

Streamlit's `@st.cache_data` decorator is used for expensive computations (filtering, sorting, fuzzy search index builds) so they don't rerun on every widget interaction. The background preloader handles initial I/O; `@st.cache_data` handles repeated in-session computations.

### Fuzzy Search

All pages with search use `rapidfuzz` rather than exact string matching:
- **Scorer:** `token_set_ratio` (handles word reordering, partial matches)
- **Threshold:** ~0.42 (tuned for legislative text which uses formal/varied phrasing)
- Search is applied after hard filters to keep the candidate set small

### Pagination

Long result lists are paginated with a configurable page size (default 25, max 250 on the Voted Bills page). Page state is stored in `st.session_state` to survive widget interactions without resetting position.

### Global Styling

`inject_css()` is called on every page to apply shared styles (fonts, card layout, color scheme) consistent with the CivicHippo brand.
